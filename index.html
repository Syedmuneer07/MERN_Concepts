React — Frontend 5 questions
01 What is the difference between functional and class components? ▶

Best Answer

Class components are ES6 classes that extend React.Component. They have lifecycle methods like componentDidMount and manage state through this.state. Functional components are plain JavaScript functions that return JSX. Before React 16.8, they were stateless — but with Hooks, they can now handle state and side effects too.

Today, functional components are the preferred approach because they're simpler, easier to test, and avoid confusing this binding issues.

// Class component
class Counter extends React.Component {
  state = { count: 0 };
  render() { return <p>{this.state.count}</p>; }
}

// Functional component (modern)
const Counter = () => {
  const [count, setCount] = useState(0);
  return <p>{count}</p>;
};
💡 Tip: Mention that hooks like useState and useEffect replaced most class lifecycle methods, making functional components the standard.
02 What are React Hooks? Explain useState and useEffect. ▶

Best Answer

Hooks are functions that let functional components "hook into" React features like state and lifecycle. They were introduced in React 16.8 to solve problems like complex class hierarchies and hard-to-reuse stateful logic.

useState adds local state to a functional component. It returns the current value and a setter function. useEffect handles side effects — things like fetching data, subscriptions, or DOM manipulation — replacing lifecycle methods like componentDidMount and componentDidUpdate.

const Example = () => {
  const [data, setData] = useState(null);

  useEffect(() => {
    // Runs after render. [] = run once on mount
    fetch('/api/data')
      .then(r => r.json())
      .then(setData);
  }, []); // dependency array

  return <div>{data?.name}</div>;
};
💡 Tip: The dependency array is key — empty [] = run once, [value] = run when value changes, no array = run every render.
03 What is a Higher Order Component (HOC)? ▶

Best Answer

A Higher Order Component is a function that takes a component and returns a new enhanced component. It's a pattern for reusing component logic — similar to higher-order functions in JavaScript.

Common use cases: authentication checks, loading states, logging, or injecting props.

const withAuth = (WrappedComponent) => {
  return (props) => {
    if (!isLoggedIn) return <Redirect to="/login" />;
    return <WrappedComponent {...props} />;
  };
};

const ProtectedDashboard = withAuth(Dashboard);
💡 Tip: You can mention that React Hooks (like custom hooks) are now often preferred over HOCs for reusing logic, but HOCs are still widely used in older codebases.
04 How does React Fiber work? ▶

Best Answer

React Fiber is the internal reconciliation engine introduced in React 16. The old engine processed the component tree synchronously in one pass — if it took too long, it would block the browser, causing dropped frames.

Fiber breaks rendering into small units of work that can be paused, resumed, and prioritized. This enables features like concurrent rendering, Suspense, and time-slicing — where React can interrupt low-priority updates to handle urgent ones (like user input) first.

💡 Tip: For a fresher, just understanding that Fiber = incremental rendering = better performance/UX is impressive enough. You don't need to know the internal linked-list structure.
05 How do you optimize performance in React apps? ▶

Best Answer

There are several key techniques:

1. React.memo — prevents re-renders if props haven't changed.
2. useMemo / useCallback — memoize expensive calculations and functions.
3. Code splitting — use React.lazy + Suspense to load components only when needed.
4. Avoid anonymous functions in JSX — they recreate on every render.
5. Virtualization — use libraries like react-window for long lists.
6. Key prop — always use stable, unique keys in lists so React can diff efficiently.

💡 Tip: Don't just list these — say "I'd first profile the app using React DevTools Profiler to identify actual bottlenecks before optimizing." This shows you think pragmatically.

06 Difference between var, let, and const. ▶

Best Answer

var

  • Function-scoped
  • Hoisted and initialized to undefined
  • Can be re-declared
  • Avoid in modern code

let

  • Block-scoped { }
  • Hoisted but NOT initialized (TDZ)
  • Cannot be re-declared
  • Use for mutable values

const

  • Block-scoped like let
  • Must be initialized at declaration
  • Cannot be reassigned (but object contents can change)
  • Prefer for most declarations
💡 Tip: Good rule of thumb — use const by default, let when you need to reassign, and never use var.
07 What is a closure? ▶

Best Answer

A closure is when an inner function retains access to variables from its outer function's scope, even after the outer function has finished executing. This is a natural result of how JavaScript's lexical scoping works.

function makeCounter() {
  let count = 0; // outer variable
  return function() {
    count++; // inner function "closes over" count
    return count;
  };
}

const counter = makeCounter();
counter(); // 1
counter(); // 2  (count persists!)
💡 Tip: Real-world uses: data privacy, memoization, event handlers, factory functions. React's useState internally uses closures.
08 Explain promises and async/await. ▶

Best Answer

A Promise is an object representing the eventual result of an async operation. It has three states: pending, fulfilled, or rejected. We chain .then() for success and .catch() for errors.

async/await is syntactic sugar over promises — it lets you write async code that looks synchronous, making it much easier to read and debug.

// Promise way
fetch('/api/user')
  .then(res => res.json())
  .then(data => console.log(data))
  .catch(err => console.error(err));

// async/await way (cleaner)
async function getUser() {
  try {
    const res = await fetch('/api/user');
    const data = await res.json();
    console.log(data);
  } catch (err) {
    console.error(err);
  }
}
💡 Tip: Always wrap await in try/catch to handle rejections. For parallel requests, use Promise.all() — it's faster than awaiting sequentially.
09 What is event bubbling and event.preventDefault()? ▶

Best Answer

Event bubbling means when an event fires on a child element, it also "bubbles up" through all its parent elements, triggering their handlers too. For example, clicking a button inside a div will also fire the div's click handler.

You can stop this with event.stopPropagation(). event.preventDefault() is different — it prevents the browser's default behavior (like a form submitting and reloading the page, or a link navigating away), but the event still bubbles.

// Prevent form page-reload
form.addEventListener('submit', (e) => {
  e.preventDefault(); // stops default behavior
  handleSubmit();
});

// Stop click from bubbling to parent
button.addEventListener('click', (e) => {
  e.stopPropagation(); // stops bubbling
});
💡 Tip: They're often confused. Simple way to remember — preventDefault() stops the browser, stopPropagation() stops other JS handlers from firing.

Node.js & Express — Backend 4 questions
10 What is middleware in Express.js? ▶

Best Answer

Middleware is a function that runs between a request arriving at the server and a response being sent. It has access to req, res, and next(). Calling next() passes control to the next middleware in the chain.

Common uses: logging, authentication, parsing request bodies, handling errors, CORS.

// Custom auth middleware
const authMiddleware = (req, res, next) => {
  const token = req.headers.authorization;
  if (!token) return res.status(401).json({ error: 'Unauthorized' });
  req.user = verifyToken(token);
  next(); // pass to next handler
};

app.get('/dashboard', authMiddleware, dashboardHandler);
💡 Tip: Think of middleware as an assembly line — each function can inspect, modify, or reject the request before it reaches the final route handler.
11 How does the request-response cycle work in Express? ▶

Best Answer

When a client sends an HTTP request, Express receives it and passes it through a chain of middleware functions in order. Each middleware can process the request, add data to it, or send a response back. If a middleware calls next(), control passes to the next one. The cycle ends when a route handler sends a response using res.send(), res.json(), or similar.

Client Request
  → Global middleware (logger, body-parser, cors)
  → Route-specific middleware (auth)
  → Route handler (business logic, DB calls)
  → Response sent to client
💡 Tip: If no middleware sends a response, the request "hangs." Always make sure every code path either calls next() or sends a response.
12 What is a REST API? ▶

Best Answer

REST (Representational State Transfer) is an architectural style for designing APIs. A REST API uses standard HTTP methods and URLs to represent and interact with resources.

  • GET     /users → fetch all users
  • POST    /users → create a user
  • PUT     /users/:id → update a user
  • DELETE /users/:id → delete a user

REST APIs are stateless — each request is self-contained and doesn't depend on previous ones. The server never stores session state between requests.

💡 Tip: Key REST principles: stateless, uniform interface, client-server separation, and resource-based URLs (nouns, not verbs — /users not /getUsers).
13 Difference between synchronous and asynchronous operations. ▶

Best Answer

Synchronous operations run one at a time, in order — each line waits for the previous to finish before executing. If one operation takes long, everything else is blocked.

Asynchronous operations don't block execution. Node.js is single-threaded, so async is essential — operations like reading files or querying a database are handed off to the event loop, freeing the thread to handle other requests while waiting.

// Sync — blocks execution
const data = fs.readFileSync('file.txt');

// Async — non-blocking
const data = await fs.promises.readFile('file.txt');
💡 Tip: In Node.js, always prefer async I/O. Sync operations in a server block the entire event loop — one slow operation can delay all users' requests.

14 What is MongoDB? ▶

Best Answer

MongoDB is a NoSQL document database that stores data as flexible JSON-like documents (BSON format) instead of rows and columns. Each document is a self-contained unit that can have nested objects and arrays.

It's schema-flexible — documents in the same collection don't need the same structure. This makes it well-suited for applications with evolving data shapes, like user profiles, product catalogs, or real-time apps.

// A MongoDB document
{
  _id: ObjectId("..."),
  name: "Alice",
  email: "alice@example.com",
  orders: [{ item: "book", qty: 2 }]  // nested!
}
💡 Tip: Mention that MongoDB is part of the MERN stack because it pairs naturally with JavaScript — data is stored and queried as JSON, eliminating the object-relational mismatch.
15 Difference between SQL and NoSQL databases. ▶

Best Answer

SQL

  • Tables with fixed schema
  • Rigid structure (rows/columns)
  • Supports JOINs across tables
  • ACID transactions
  • MySQL, PostgreSQL

NoSQL

  • Documents, key-value, graphs
  • Flexible, schema-less
  • Data embedded in documents
  • Scales horizontally
  • MongoDB, Redis, Cassandra
💡 Tip: Neither is "better" — SQL is great for relational data with complex queries; NoSQL shines for flexible, high-volume, rapidly evolving data. Show you understand the trade-offs.
16 What is aggregation in MongoDB? ▶

Best Answer

Aggregation is MongoDB's way to process and transform documents using a pipeline — a series of stages where the output of one stage becomes the input of the next. It's similar to SQL's GROUP BY, JOIN, and HAVING combined.

// Count orders per user, only show top 5
db.orders.aggregate([
  { $match: { status: "completed" } },  // filter
  { $group: { _id: "$userId", total: { $sum: 1 } } }, // group
  { $sort: { total: -1 } },  // sort descending
  { $limit: 5 }  // take top 5
]);
💡 Tip: Common stages to know: $match (filter), $group (aggregate), $project (select fields), $sort, $limit, $lookup (join).
17 How do you design a schema in MongoDB? ▶

Best Answer

MongoDB is schema-less by nature, but in practice we define structure using Mongoose schemas in Node.js. The key design decision is embedding vs. referencing:

Embed when data is always accessed together and the nested data is small (e.g., a user's address inside the user document). Reference (like a foreign key) when data is large, shared across documents, or updated independently (e.g., orders referencing a product by ID).

// Mongoose schema with Mongoose
const userSchema = new Schema({
  name: { type: String, required: true },
  email: { type: String, unique: true },
  age: { type: Number, min: 0 },
  createdAt: { type: Date, default: Date.now }
});
💡 Tip: Design schema based on query patterns, not just data relationships. How you'll read the data should drive how you structure it.

18 Build a simple counter app in React. ▶

Clean Solution

import { useState } from 'react';

export default function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <h2>Count: {count}</h2>
      <button onClick={() => setCount(c => c + 1)}>+</button>
      <button onClick={() => setCount(c => c - 1)}>−</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  );
}
💡 Tip: Use the functional updater form c => c + 1 instead of count + 1 — it's safer when state updates are batched (shows you understand React's async state).
19 Create a form with validation in React. ▶

Clean Solution

function LoginForm() {
  const [form, setForm] = useState({ email: '', password: '' });
  const [errors, setErrors] = useState({});

  const validate = () => {
    const e = {};
    if (!form.email.includes('@')) e.email = 'Invalid email';
    if (form.password.length < 6) e.password = 'Min 6 characters';
    return e;
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    const errs = validate();
    if (Object.keys(errs).length) { setErrors(errs); return; }
    console.log('Submitted', form);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input value={form.email}
        onChange={e => setForm({...form, email: e.target.value})} />
      {errors.email && <p>{errors.email}</p>}
      <button type="submit">Login</button>
    </form>
  );
}
💡 Tip: For production, mention libraries like React Hook Form or Formik with Yup for complex validation — shows real-world awareness.
20 Fetch and display data from an API. ▶

Clean Solution

function UserList() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch('https://jsonplaceholder.typicode.com/users')
      .then(r => { if (!r.ok) throw new Error('Failed'); return r.json(); })
      .then(setUsers)
      .catch(setError)
      .finally(() => setLoading(false));
  }, []);

  if (loading) return <p>Loading...</p>;
  if (error) return <p>Error: {error.message}</p>;
  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}
💡 Tip: Always handle all three states: loading, error, and success. Interviewers watch for this. Also add !r.ok check — fetch doesn't throw on 4xx/5xx errors.
21 Implement a basic CRUD operation (Express + MongoDB). ▶

Clean Solution

// Express CRUD routes for a User resource
const router = express.Router();

// CREATE
router.post('/', async (req, res) => {
  const user = await User.create(req.body);
  res.status(201).json(user);
});

// READ ALL
router.get('/', async (req, res) => {
  const users = await User.find();
  res.json(users);
});

// UPDATE
router.put('/:id', async (req, res) => {
  const user = await User.findByIdAndUpdate(req.params.id, req.body, { new: true });
  res.json(user);
});

// DELETE
router.delete('/:id', async (req, res) => {
  await User.findByIdAndDelete(req.params.id);
  res.status(204).send();
});
💡 Tip: Wrap each handler in try/catch and return appropriate HTTP status codes: 201 for created, 204 for deleted, 404 for not found. This shows production thinking.
